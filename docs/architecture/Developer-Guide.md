# Developer Guide

**Project:** SWGOH Guild Suite

Version: 0.1 (Draft)

---

# 1. Purpose

SWGOH Guild Suite is an open-source toolkit for Star Wars: Galaxy of Heroes.

The project began as a personal tool for managing the guild **BuzzLighter**, but every architectural decision must allow it to scale into a universal toolkit for any guild.

The goal is not to create another Discord bot.

The goal is to build a reusable platform where multiple independent tools share the same data model.

---

# 2. Core Principles

## Single Source of Truth

Every piece of information must exist in only one canonical place.

Example:

- GuildSnapshot describes the current guild state.
- GameData describes the game.
- Scenario describes an event.
- Reports never duplicate data.

---

## Read Only

The project never performs actions on player accounts.

No login.

No automation.

No gameplay interaction.

Only publicly available information or data explicitly provided by users.

---

## Modular Architecture

Every subsystem must work independently.

Current planned modules:

- Snapshot Builder
- Scenario Builder
- Assignment Engine
- Report Generator
- Guild Analytics
- Gear Planner

Modules communicate only through shared data models.

No module should directly depend on another module's implementation.

---

## API Isolation

External APIs are adapters.

The project must never expose Comlink objects internally.

Comlink → Adapter → Internal Models

If Comlink changes tomorrow, only the adapter should require updates.

---

## Data before UI

Business logic always comes first.

Every algorithm must work without a graphical interface.

The UI is only a visualization layer.

---

## Human First

The goal is to assist guild officers.

The system suggests decisions.

The officer makes the final decision.

Automatic assignment must always remain editable.

# 4. Internal Data Models

The project is built around a small number of canonical models.

## GameData

Static information about the game.

Examples:

- units
- abilities
- gear
- events
- currencies

---

## GuildSnapshot

Live snapshot of one guild.

Contains:

- guild information
- players
- player rosters
- activity
- GP

Generated from Comlink.

---

## Inventory

Player-owned resources.

Provided only by the player.

Never available through Comlink.

---

## Scenario

Description of an event.

Examples:

- Territory Battle
- Territory War
- Raid

A Scenario never contains player data.

---

## Assignment

Output of optimization algorithms.

Examples:

- Platoon assignments
- TW defense plan
- Gear farming plan

Assignments can always be regenerated.

# 8. Coding Guidelines

- Keep functions small.
- Prefer composition over inheritance.
- Avoid duplicated logic.
- Prefer explicit names.
- Every module should be testable independently.
- JSON files should be human-readable.
